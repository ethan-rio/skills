How does our rate limiter actually work?

A from-scratch tour for someone who has touched the limiter once and wants the mental model — token bucket, Redis, and the two foot-guns that bit us last quarter.

TL;DR

One shared Redis-backed token bucket per (user × route-group). The Lua script atomically refills, deducts, and returns the new count. App code only decides "did I get a token?". Two gotchas: clock skew across app servers, and the silent fail-open path on Redis errors.

1 · The flow at a glance

2 · The Lua script

Atomicity is the whole reason this lives in Redis. App servers compute nothing about the bucket — they just hand keys and parameters to Lua and read back a boolean.

-- KEYS[1] = bucket key, ARGV = now_ms, capacity, refill_per_sec, take local bucket = redis.call('HMGET', KEYS[1], 'tokens', 'ts') local cap, refill, take = tonumber(ARGV[2]), tonumber(ARGV[3]), tonumber(ARGV[4]) local now = tonumber(redis.call('TIME')[1]) * 1000 -- <-- not ARGV[1]; clock-safe local tokens, ts = tonumber(bucket[1]) or cap, tonumber(bucket[2]) or now local elapsed = math.max(0, (now - ts) / 1000) tokens = math.min(cap, tokens + elapsed * refill) local ok = 0 if tokens >= take then tokens = tokens - take; ok = 1 end redis.call('HMSET', KEYS[1], 'tokens', tokens, 'ts', now) redis.call('PEXPIRE', KEYS[1], 60000) return { ok, tokens }
// per-request, after authn key := userID + ":" + routeGroup ok, _, err := limiter.Eval(ctx, key, group.Capacity, group.RefillPerSec, 1) if err != nil { metrics.LimiterErr.Inc() return nextHandler() // fail-open — see callout } if !ok { return c.JSON(429, retryAfter(group)) } return nextHandler()
groups: api: { capacity: 60, refill_per_sec: 1 } search: { capacity: 30, refill_per_sec: 0.5 } upload: { capacity: 5, refill_per_sec: 0.1 }

3 · Walk-through: what happens on a request

Step 1 · middleware.go computes the bucket key

The key is rl:<userID>:<group> for authenticated users, or rl:ip:<ip>:<group> for anonymous. Group comes from routes.yaml; if a route is missing a group it falls into api by default.

Step 2 · Redis runs limiter.lua atomically

Lua reads the bucket hash, computes elapsed time using Redis's clock (this is the fix for the bug that bit us — see §5), refills, deducts 1, persists, and returns (ok, remaining).

Step 3 · App returns 429 or proceeds

On ok=0 we return 429 with a Retry-After header derived from refill rate. On ok=1 we attach X-RateLimit-Remaining and continue.

Step 4 · Bucket TTL and cleanup

Every Lua call PEXPIREs the key for 60s. So idle users' buckets disappear automatically — no GC job needed.

4 · Foot-guns

Gotcha 1 · clock skew Pre-2026-02 the script took now from the calling app server. Two app servers with skewed NTP would alternately advance and rewind the bucket's ts for the same key. Symptom: users seeing burst-then-stall behavior. Fix: redis.call('TIME') inside Lua. Always.
Gotcha 2 · fail-open silently Redis transient errors return nextHandler(). That means a Redis blip silently disables the limiter. We accept this trade-off (availability over correctness) but we instrument limiter.err and alert on sustained nonzero. Don't remove the alert.

5 · FAQ

Is this distributed rate limiting?
Yes — Redis is the single source of truth. Any number of app servers, any number of pods, single bucket per user×group.
What about bursts during deploys?
Token bucket is bursty by design (capacity = burst budget). If you don't want bursts, switch to a sliding window — see #4127.
Why not RATELIMIT from Redis modules?
We can't deploy modules on managed Redis. The 12-line Lua we own is fine.
token bucket
An algorithm where tokens accumulate at a fixed rate, capped at a capacity; each request takes a token; rejects when empty.
fail-open
When the limiter errors, requests pass through unrestricted. Opposite: fail-closed (reject all).
PEXPIRE
Redis command to set a millisecond TTL on a key.
