Offline sync for collaborative notes

A spec for letting users edit notes offline and converging on a clean shared state when they reconnect — without losing edits, surprising the user, or shipping a CRDT we'll regret in two years.

1 · Problem

Currently the editor requires a live websocket. Every keystroke flies to the server. On flaky cellular or in airplane mode, users see a frozen UI and lose work. We've had four support escalations this month.

2 · Constraints

  • Must converge — never silently drop a user's edit
  • Must work for documents up to ~100k chars (95th percentile is ~12k)
  • Must respect the existing document_versions table; no schema rewrite
  • Must ship behind the offline_sync flag, dark-launchable

3 · Three approaches

Option A

Last-write-wins with operation log

Queue ops locally, replay on reconnect, server timestamps win.

  • Two days to ship
  • No new dependency
  • Concurrent edits clobber each other
  • Users notice

Best when: solo-edit workflows dominate (they don't, for us)

Option B · recommended

Yjs CRDT with IndexedDB persistence

Battle-tested CRDT, automatic merge, plugs into ProseMirror.

  • Real merge, no data loss
  • Existing y-prosemirror binding
  • ~80kb gzip added
  • Document size grows ~1.4×

Best when: collab is core. It is.

Option C

Custom OT with server arbiter

Operational transform like Google Docs of yore.

  • Smaller payload than CRDT
  • Six months to ship right
  • Edge cases haunt you
  • Nobody on the team has done OT

Best when: never, for a 4-person team

Decision  Going with Option B. Yjs has the best maturity per engineering hour we'll spend.

6 · Open questions

Open  What's our retention policy for orphaned IndexedDB docs? (User logs out on a shared machine.) Default to wipe on logout, but confirm with security.
Open  Do we GC the server's update log? Yjs grows monotonically. ~6 weeks before 100MB at p95.

7 · Out of scope

  • Mobile app (separate spec, after web ships)
  • Cross-document references / transclusion
  • Read receipts / cursor presence (presence already works; we keep it)
